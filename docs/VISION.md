# GitHub Insights - 設計思想

## なぜ作るのか

GitHub上の貢献データは分散している。

| 場所 | 問題 |
|------|------|
| コントリビューショングラフ | 活動量はわかるが、中身が伝わらない |
| リポジトリページ | 個別の貢献は見えるが、全体像がない |
| プロフィールページ | 情報が点在し、インパクトが伝わりにくい |

この分散により：
- 自分自身でも全体像を把握しにくい
- 日々の活動が成果として認識しにくい
- 他者（採用担当、コミュニティ）へのアピールが難しい

**GitHub Insights は、分散したデータを集約し、貢献度を伝わる形で可視化する。**

---

## 設計原則

### 1. 達成感のデザイン

コミット数などの数値を、バッジやスコアといった具体的な評価に変換する。

| Before | After |
|--------|-------|
| コミット履歴の羅列 | 「Core Contributor」などのバッジ |
| 数値の羅列 | パーセンタイル表示（上位 X%） |

数値を眺めるだけでは得られない達成感を提供する。

### 2. 共有体験のデザイン

成果を外部に発信しやすくする。

- **OGカード生成**: URLを共有するだけで、視覚的なサマリーが表示される
- **Xシェアボタン**: ワンクリックでSNS共有
- **GitHub Wrapped**: 年間統計を共有可能な形式で提供

### 3. 洞察の可視化

データから新たな発見を得られるようにする。

| 機能 | 提供する洞察 |
|------|------------|
| コントリビューターランキング | 特定リポジトリでの自分の位置づけ |
| 言語統計 | プロジェクトの技術構成 |
| コミット推移 | 開発活動のトレンド |
| ヒートマップ | 活動パターン（時間帯・曜日） |

### 4. 信頼の基盤設計

最小権限の原則を徹底する。

| 要求する権限 | 要求しない権限 |
|-------------|--------------|
| コードの読み取り | リポジトリへの書き込み |
| メールアドレスの読み取り | 設定の変更 |
| プロフィール情報の読み取り | その他の書き込み操作 |

**リポジトリへの書き込み権限は一切要求しない。** ユーザーのコードが変更されるリスクを原理的に排除する。

---

## まとめ

GitHub Insights が提供する価値：

1. **分散したデータの集約** → 全体像の把握
2. **数値から評価への変換** → 達成感の可視化
3. **共有しやすい形式** → 外部へのアピール
4. **最小権限の設計** → 安心して利用できる
